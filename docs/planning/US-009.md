## User Story

**Epic:** EP-002 — Quotazioni Near-Real-Time
**Priorità:** MEDIUM | **Story Points:** 3

**Story**
As Marco (investitore retail),
I want che il sistema conservi le ultime quote recuperate,
so that la dashboard sia sempre visivamente popolata anche quando il provider non risponde immediatamente.

**Criteri di Accettazione**
- [ ] Ogni quota recuperata dal provider viene salvata nella tabella QuoteCache con ticker, prezzo, variazione, timestamp
- [ ] Al caricamento della dashboard, se il provider non risponde entro 3 secondi, vengono mostrati i dati dalla cache
- [ ] I dati dalla cache sono marcati visivamente come "stale" con il timestamp dell'ultimo aggiornamento reale
- [ ] La cache viene sovrascritta ad ogni refresh riuscito dal provider

## Analisi dei Requisiti

> **Analista:** Emanuele 🔎

### Scope

**Incluso:**
- Salvataggio automatico delle quote nella tabella `QuoteCache` dopo ogni fetch riuscito dal provider
- Endpoint dedicato per lettura diretta dalla cache (`GET /api/quotes/cache`)
- Fallback automatico alla cache con timeout di 3 secondi sul provider
- Indicazione visiva "stale" con timestamp dell'ultimo aggiornamento reale
- Sovrascrittura cache ad ogni refresh riuscito

**Escluso:**
- TTL o scadenza automatica della cache (la cache è sempre valida come fallback)
- Cache invalidation manuale da parte dell'utente
- Cache di dati storici (solo quote correnti)

### Requisiti Impliciti

- Il modello `QuoteCache` esiste già nello schema Prisma con: id, symbolId (unique), price, change, changePercent, high, low, volume, timestamp, updatedAt
- L'upsert sulla cache deve usare `symbolId` come chiave unica
- Il frontend deve distinguere tra dati "fresh" (dal provider) e dati "stale" (dalla cache)
- Il timeout di 3 secondi si applica alla chiamata al provider, non alla risposta totale dell'API

### Assunzioni

- US-006 (batch quotes) è completato — l'endpoint `/api/quotes/batch` esiste
- US-007 (refresh automatico) è completato o in parallelo — il polling riusa lo stesso meccanismo
- Il `QuoteCache` ha relazione con `Symbol` via `symbolId`
- La cache viene popolata solo dopo un fetch riuscito (non con dati parziali o errati)

## Soluzione Tecnica

> **Architetto:** Leonardo 📐

### Approccio Scelto

Strategia cache-aside con write-through: ogni fetch riuscito dal provider salva i dati in `QuoteCache` via Prisma upsert. Al caricamento della dashboard, il frontend richiede prima i dati dalla cache per un "fast initial paint", poi lancia il fetch dal provider. L'endpoint batch viene modificato con un timeout di 3 secondi e fallback automatico alla cache.

### Motivazione

Il pattern cache-aside è semplice e affidabile. Usare il database come cache (anziché Redis) è coerente con lo stack esistente e sufficiente per il volume di dati previsto. Il fallback con timeout garantisce che la dashboard sia sempre popolata senza rallentamenti percepiti dall'utente.

### Modifiche al Data Model

Nessuna — il modello `QuoteCache` esiste già con tutti i campi necessari:

```prisma
model QuoteCache {
  id            String   @id @default(uuid())
  symbolId      String   @unique
  price         Float
  change        Float
  changePercent Float
  high          Float?
  low           Float?
  volume        Float?
  timestamp     DateTime
  updatedAt     DateTime @updatedAt
  symbol        Symbol   @relation(fields: [symbolId], references: [id], onDelete: Cascade)
}
```

### Modifiche alle API

**`GET /api/quotes/batch` (modifica):**
- Dopo un fetch riuscito dal provider, salvare ogni quota in `QuoteCache` via `prisma.quoteCache.upsert()`
- Implementare timeout di 3 secondi con `Promise.race()` o `AbortController`
- Se il provider non risponde entro 3s, leggere dalla `QuoteCache` per i simboli richiesti
- Aggiungere campo `source: "provider" | "cache"` nella risposta
- Aggiungere campo `cachedAt: string` (ISO timestamp) quando la source è "cache"

**`GET /api/quotes/cache?symbols=AAPL,MSFT` (nuovo):**
- Nessuna auth richiesta (segue pattern esistente)
- Parametro query `symbols`: lista ticker separati da virgola
- Risposta 200: `{ quotes: QuoteCacheItem[], source: "cache" }`
- Risposta 400: parametro symbols mancante
- Se nessun dato in cache, restituire array vuoto (non errore)

### Modifiche al Frontend

**WatchlistPanel:**
- Al mount, fare fetch da `/api/quotes/cache?symbols=...` per "fast initial paint"
- Immediatamente dopo (o in parallelo), lanciare fetch da `/api/quotes/batch`
- Se i dati provengono dalla cache (`source === "cache"`), mostrare indicatore "stale":
  - Badge/tooltip "Dati aggiornati alle HH:MM" con il timestamp della cache
  - Opacità ridotta o bordo tratteggiato per differenziare visivamente
- Quando arrivano dati fresh dal provider, sostituire i dati stale e rimuovere l'indicatore
