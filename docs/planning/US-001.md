## User Story

**Epic:** EP-001 — Ricerca Simboli e Watchlist
**Priorità:** HIGH | **Story Points:** 5

**Story**
As investitore retail o studente di finanza,
I want che il sistema abbia i modelli dati e l'integrazione con il provider di mercato pronti,
so that tutte le funzionalità successive (watchlist, quote, alert) possano funzionare su una base solida.

**Criteri di Accettazione**
- [ ] Lo schema Prisma include i modelli Watchlist, WatchlistItem, Symbol (ticker, name, exchange), Alert e QuoteCache con relazioni corrette e scoping per utente
- [ ] L'adapter market-data espone un'interfaccia astratta (provider.ts) con implementazione Twelve Data (twelve-data.ts)
- [ ] Il normalizer (quote-normalizer.ts) converte la risposta Twelve Data in un formato interno coerente con prezzo, variazione, percentuale e timestamp
- [ ] Una route API di test (/api/quotes?symbol=AAPL) restituisce una quota normalizzata dal provider
- [ ] La API key Twelve Data è conservata solo lato server e mai esposta al client

## Analisi dei Requisiti

> **Analista:** Emanuele 🔎

### Scope

- Estendere lo schema Prisma attuale (che ha solo il modello `User`) con 5 nuovi modelli: `Watchlist`, `WatchlistItem`, `Symbol`, `Alert`, `QuoteCache`
- Creare un adapter layer in `src/lib/market-data/` con interfaccia astratta e implementazione Twelve Data
- Creare un normalizer che trasformi la risposta grezza di Twelve Data in un formato interno coerente
- Esporre una route API di test `GET /api/quotes?symbol=AAPL` che dimostri il funzionamento end-to-end
- La API key Twelve Data deve vivere solo lato server come variabile d'ambiente (mai `NEXT_PUBLIC_`)

### Requisiti Impliciti

- Le relazioni tra modelli devono prevedere scoping per utente (`userId` su Watchlist e Alert) per garantire isolamento dati
- Il modello `Symbol` deve essere condiviso (non per-utente) per evitare duplicazione dati ticker
- `QuoteCache` deve essere indicizzato per ticker per lookup rapido
- L'interfaccia astratta del provider deve prevedere almeno il metodo per ottenere una quota (e in prospettiva ricerca simboli e storico)
- L'env var `TWELVE_DATA_API_KEY` va documentata e usata solo nei Route Handler server-side
- Gestione errori base nel provider (API non raggiungibile, risposta malformata, rate limit)
- Il normalizer deve gestire gracefully risposte incomplete senza crash

### Assunzioni

- Il free tier di Twelve Data supporta l'endpoint `/quote` per titoli USA
- Non è necessario implementare WebSocket o polling in questa story (saranno in US-007)
- La route `/api/quotes` è un endpoint di verifica; la UI consumer arriverà con US-006

### Modifiche al Data Model

**Nuovi modelli:**

- `Symbol` — ticker (unique), name, exchange. Entità condivisa globalmente. Relazioni verso WatchlistItem, Alert, QuoteCache.
- `Watchlist` — name, userId (FK → User). Scoping per utente. Relazione 1:N con WatchlistItem.
- `WatchlistItem` — watchlistId (FK → Watchlist), symbolId (FK → Symbol), sortOrder. Vincolo unique su (watchlistId, symbolId).
- `Alert` — userId (FK → User), symbolId (FK → Symbol), direction ("above"/"below"), threshold, status ("active"/"triggered"/"disabled"), triggeredAt, triggeredPrice.
- `QuoteCache` — symbolId (FK → Symbol, unique), price, change, changePercent, high, low, volume, timestamp.

**Modifica modello esistente:**

- `User` — aggiunta relazioni inverse `watchlists Watchlist[]` e `alerts Alert[]`.

### Modifiche alle API

| Endpoint | Metodo | Descrizione |
|---|---|---|
| `/api/quotes?symbol=AAPL` | GET | Restituisce una quota normalizzata dal provider. Parametro `symbol` obbligatorio. |

Risposte:
- `200` — `{ symbol, name, exchange, price, change, changePercent, high, low, volume, timestamp }`
- `400` — `{ error: "Symbol parameter is required" }`
- `502` — `{ error: "Failed to fetch quote from provider" }`

### Modifiche al Frontend

Nessuna modifica al frontend in questa story. La route API è di test/verifica.
