## User Story

**Epic:** EP-004 — Alert di Prezzo
**Priorità:** HIGH | **Story Points:** 3

**Story**
As Marco (investitore retail),
I want impostare un alert quando il prezzo di un titolo supera o scende sotto una soglia che scelgo,
so that possa essere avvisato di movimenti rilevanti senza controllare continuamente.

**Criteri di Accettazione**
- [ ] L'utente può creare un alert specificando: titolo, direzione (sopra/sotto) e soglia numerica
- [ ] L'alert viene salvato su database e associato all'utente autenticato
- [ ] L'utente può creare più alert per lo stesso titolo con soglie diverse
- [ ] L'alert viene valutato ad ogni refresh delle quote e si attiva quando la condizione è soddisfatta
- [ ] L'utente può disattivare o eliminare un alert esistente

## Analisi dei Requisiti

> **Analista:** Emanuele 🔎

### Scope

**Incluso:**
- API CRUD per gli alert (create, list, delete, update status)
- Form per creare un alert: selezionare titolo, direzione (above/below), soglia
- Logica di valutazione alert ad ogni refresh quote
- Gestione stati alert: active, triggered, disabled
- UI per visualizzare, disattivare ed eliminare alert

**Escluso:**
- Evidenza alert attivati in dashboard (US-013)
- Notifiche email (US-017)

### Requisiti Impliciti

- Il modello Alert esiste già nello schema Prisma con direction, threshold, status, triggeredAt, triggeredPrice
- La valutazione alert avviene client-side durante il polling delle quote (hook useQuotePolling)
- Quando un alert viene triggato: update status=triggered, triggeredAt=now, triggeredPrice=currentPrice
- Un alert triggered non viene rivalutato (one-shot)
- L'utente deve vedere il prezzo corrente del titolo quando crea l'alert (per orientarsi sulla soglia)

### Assunzioni

- US-007 (polling quote) è completato
- Gli alert sono valutati lato client con una chiamata API per il trigger
- Lo schema Alert è già pronto (direction: string "above"/"below", status: "active"/"triggered"/"disabled")

## Soluzione Tecnica

> **Architetto:** Leonardo 📐

### Approccio Scelto

API REST per CRUD alert + logica di valutazione nel polling hook + componente AlertForm per la creazione e AlertList per la gestione.

### Motivazione

La valutazione alert durante il polling evita un servizio separato e sfrutta i dati già fetchati. Le API REST seguono il pattern esistente. Il modello Alert è già nel DB.

### Modifiche al Data Model

Nessuna modifica — il modello Alert esiste già con tutti i campi necessari.

### Modifiche alle API

**`GET /api/alerts`** — Lista alert dell'utente autenticato
- Risposta 200: `{ alerts: Alert[] }` (con symbol incluso)

**`POST /api/alerts`** — Crea nuovo alert
- Body: `{ symbolId: string, direction: "above" | "below", threshold: number }`
- Risposta 201: `{ alert: Alert }`

**`PATCH /api/alerts/[id]`** — Aggiorna stato alert
- Body: `{ status: "active" | "disabled" }` o `{ status: "triggered", triggeredPrice: number }`
- Risposta 200: `{ alert: Alert }`

**`DELETE /api/alerts/[id]`** — Elimina alert
- Risposta 200: `{ success: true }`

**`POST /api/alerts/evaluate`** — Valuta alert con quote correnti
- Body: `{ quotes: Record<string, { price: number }> }`
- Logica: per ogni alert active, controlla se la condizione è soddisfatta
- Risposta 200: `{ triggered: Alert[] }` (alert appena triggerati)

### Modifiche al Frontend

**`AlertForm`:** Dialog/modale per creare un alert su un titolo. Mostra prezzo corrente, input soglia, selettore direzione (sopra/sotto).

**`AlertList`:** Lista degli alert attivi con stato, soglia, direzione. Azioni: disattivare, eliminare.

**Polling integration:** Dopo ogni refresh quote, chiamare `/api/alerts/evaluate` con i prezzi correnti per triggerare alert.
