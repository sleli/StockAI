As Giulia (studentessa di finanza),
I want ritrovare rapidamente i titoli che ho visualizzato di recente,
so that possa riprendere l'osservazione senza dover cercare di nuovo.

**Criteri di Accettazione**
- [ ] Il sistema registra gli ultimi titoli visualizzati dall'utente (massimo 10)
- [ ] Una sezione "Visti di recente" è visibile nella dashboard o nel pannello di ricerca
- [ ] I titoli recenti sono cliccabili e portano direttamente alla pagina dettaglio
- [ ] La lista si aggiorna automaticamente quando l'utente visita un nuovo titolo

**Incluso:**

- Registrazione automatica della visita alla pagina `/stocks/[symbol]`

**Escluso:**
- Persistenza lato client (localStorage) — si usa il DB per coerenza cross-device
- Cronologia completa delle visualizzazioni (solo le ultime 10)
- Suggerimenti basati sulla cronologia (feature futura)

### Requisiti Impliciti

- Serve un modello `RecentView` con relazione all'utente e al simbolo del titolo
- L'API POST deve gestire il caso di titolo già presente: aggiornare il timestamp invece di duplicare
- L'API GET deve restituire i titoli ordinati per data di visualizzazione (più recente prima)
- La registrazione della visita deve avvenire solo per utenti autenticati
- Il componente deve mostrare ticker, nome e prezzo corrente se disponibile

### Assunzioni

- La pagina `/stocks/[symbol]` esiste già (US-010)
- L'utente deve essere autenticato per registrare le visualizzazioni
- Il limite di 10 è gestito lato server (eliminazione del più vecchio quando si supera il limite)
- I dati del titolo (nome, prezzo) vengono recuperati al momento della visualizzazione del componente

## Soluzione Tecnica

> **Architetto:** Leonardo 📐

### Approccio Scelto

Nuovo modello Prisma `RecentView`, API REST `/api/recent-views` con POST (registra) e GET (lista), componente `RecentlyViewed` nella dashboard, hook di registrazione automatica nella pagina dettaglio titolo.

### Motivazione

Persistenza server-side per coerenza cross-device e cross-sessione. Il modello Prisma con unique constraint su (userId, symbol) garantisce nessun duplicato — la visita ripetuta aggiorna solo il timestamp. Il limite di 10 è gestito con query `take: 10` + cleanup periodico.

### Modifiche al Data Model

**Nuovo modello `RecentView`:**

```prisma
model RecentView {
  id        String   @id @default(uuid())
  userId    String
  symbol    String
  name      String?
  viewedAt  DateTime @default(now()) @updatedAt
  user      User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([userId, symbol])
  @@index([userId, viewedAt(sort: Desc)])
}
```

- `@@unique([userId, symbol])` — impedisce duplicati, permette upsert
- `@@index([userId, viewedAt])` — ottimizza la query ordinata per data
- `onDelete: Cascade` — pulizia automatica quando l'utente viene eliminato

### Modifiche alle API

**`POST /api/recent-views`**
- Auth richiesta (401 se non autenticato)
- Body: `{ symbol: string, name?: string }`
- Logica: upsert su (userId, symbol), aggiorna `viewedAt`
- Dopo upsert: eliminare record eccedenti (> 10 per utente)
- Risposta 200: `{ success: true }`
- Risposta 400: symbol mancante
- Risposta 401: non autenticato

**`GET /api/recent-views`**
- Auth richiesta (401 se non autenticato)
- Risposta 200: `{ views: RecentView[] }` (max 10, ordinati per viewedAt desc)
- Risposta 401: non autenticato

### Modifiche al Frontend

**Componente `RecentlyViewed`:**
- Client component che fa fetch di `/api/recent-views` al mount
- Lista orizzontale o griglia compatta di card/chip con ticker e nome
- Ogni elemento è un link a `/stocks/[symbol]`
- Stato vuoto: messaggio "Nessun titolo visualizzato di recente"
- Loading skeleton durante il fetch

**Pagina `/stocks/[symbol]` — registrazione automatica:**
- useEffect che chiama POST `/api/recent-views` con symbol e nome del titolo
- Fire-and-forget: non blocca il rendering, errori silenziati
- Solo per utenti autenticati (check prima della chiamata)
